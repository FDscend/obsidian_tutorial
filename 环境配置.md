# git

## 配置

（弹窗登录 Github），还要手动配置账户

```bash
git config --global user.name "John Doe"
git config --global user.email johndoe@example.com
```

网络代理（填自己的端口号）

```bash
git config --global http.proxy http://127.0.0.1:1234
git config --global https.proxy https://127.0.0.1:1234
```

## 克隆仓库

### 默认克隆

```bash
git clone https://github.com/FDscend/obsidian_tutorial.git
```

会在终端主目录下创建 `obsidian_tutorial` 文件夹，仓库展开放到里面

### 新文件夹名

```bash
git clone https://github.com/FDscend/obsidian_tutorial.git ObsidianTutorial
```

会在终端主目录下创建 `ObsidianTutorial` 文件夹，仓库展开放到里面（建议在终端中使用）

### 当前文件夹

```bash
git clone https://github.com/FDscend/obsidian_tutorial.git .
```

（注意标点 `.`）仓库直接展开放到主目录下，建议在 vscode 中使用

### 选择分支

1. 准备工作：关闭 VS Code。在文件管理器中，确保你的 `ObsidianTutorial` 文件夹是**空的**。如果里面有之前尝试产生的 `obsidian_tutorial` 子文件夹或其他东西，最好先删除它们，从一个干净的空文件夹开始。
2. 在 `ObsidianTutorial` 文件夹里**右键点击**，选择用 VS Code 打开
3. 在 VS Code 的终端中，输入克隆命令（注意标点 `.`）

   ```bash
   git clone https://github.com/FDscend/obsidian_tutorial.git .
   ```

执行完这个命令后，`ObsidianTutorial` 文件夹里出现的就直接是 `main` 分支的内容了。

4. 点击左下角 `main` 选择切换分支
5. 在源代码管理侧栏的“图形”，点击 `main` 选择切换分支，之后只展示该分支的更新

## 提交

在 vscode 的源代码管理：

1. 选择需要更新的文件，点击**暂存更改**
2. 填写更改原因
3. 点击**提交**
4. 点击**同步更改**

> 你可能同时修改了多个文件，但这次提交（Commit）你只想提交其中相关的部分。暂存区允许你**精确选择哪些修改要纳入下一次提交**。
>
> 你同时修改了  `slide1.md`（修复了一个拼写错误）和  `slide2.md`（添加了新的内容），这是两个独立的修改，如果
>
> - **没有暂存区**：你只能一次性提交所有文件的更改，提交信息会变得混乱（例如：“修复拼写错误并添加新内容”）
> - **有暂存区**：可以只暂存  `slide1.md`  的更改，然后提交，信息为“修复 slide1 的拼写错误”。再暂存  `slide2.md`  的更改，然后提交，信息为“在 slide2 添加新内容部分”

## 合并冲突

> [!faq]
>
> 如果其他人编辑了内容，我也修改了内容，那么在我提交的时候可以会有冲突，怎么办？

Git 处理冲突的过程非常清晰，可以概括为：

```mermaid
graph LR
  A("尝试自动合并")-->B("发现冲突")
  B-->C("暂停并标记冲突")
  C-->D("等待用户解决")-->E("合并")
```

### 第 1 步：你尝试推送更改，但被拒绝

当你完成本地修改并执行 `git push` 时，如果这期间有其他人已经推送了他们的提交，你会收到类似这样的错误：

```
! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'github.com:...'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. This is usually caused by another repository pushing
hint: to the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
```

这条消息的意思是：**你的本地历史已经落后于远程仓库，Git 拒绝推送以避免覆盖他人的工作**。它建议你先整合远程的更改。

### 第 2 步：你拉取远程更改，Git 尝试自动合并

你听从建议，执行 `git pull`（它相当于 `git fetch` + `git merge`）。

这时，Git 会做两件事：

1.  **下载**（fetch）远程仓库的最新提交到你的本地。
2.  **尝试合并**（merge）远程的更改和你的本地更改。

- **最佳情况（无冲突）**：如果你和同事修改了**不同的文件**，或者修改了同一个文件的**不同部分**，Git 非常智能，它可以**自动完成合并**。它会创建一个新的“合并提交”（Merge Commit），把你和同事的更改融合在一起。这时你直接 `git push` 即可。
- **最坏情况（有冲突）**：如果你和同事修改了**同一个文件的同一个区域**，Git 无法判断到底应该保留谁的更改。它会**自动合并失败**并进入冲突状态。

### 第 3 步：Git 标记冲突，等待解决

如果合并冲突，Git 不会让你不知所措。它会在那些有冲突的文件里做上**明确的标记**，告诉你哪里出了问题。

一个被标记了冲突的文件看起来是这样的：

```
# 我的数学幻灯片
<<<<<<< HEAD
这是我今天添加的非常重要的一段新内容。
=======
这是我同事修改和优化的同一段内容。
>>>>>>> a1b2c3d4... (这里是同事提交的哈希值和信息)

幻灯片的一些其他内容。
```

- **`<<<<<<< HEAD`** 到 **`=======`** 之间的内容，是**你当前的本地更改**（在 `HEAD` 指向的提交里）。
- **`=======`** 到 **`>>>>>>> ...`** 之间的内容，是**远程拉下来的、别人做的更改**（来自那个叫 `a1b2c3d4...` 的提交）。
- Git 把选择权完全交给了你。

### 第 4 步：你手动解决冲突

现在你需要做决定：保留你的版本？保留同事的版本？还是创造一个全新的、结合两人智慧的新版本？

1.  **编辑文件**：用 VS Code 打开这个文件。
2.  **做出选择**：你需要**手动删除所有这些 `<<<<<<<`， `=======`， `>>>>>>>` 标记**，并把内容修改成你最终希望的样子。
    - 例如，你可以把它改成：

      ```
      # 我的数学幻灯片

      这是我们共同商定后，合并在一起的最佳内容版本。

      幻灯片的一些其他内容。
      ```

3.  **VS Code 的强大工具**：VS Code 对此有非常好的可视化支持。你可以在冲突区域看到“Accept Current Change”（接受你的更改）、“Accept Incoming Change”（接受同事的更改）、“Accept Both Changes”（接受双方更改）等选项按钮，点击即可自动解决，无需手动删除标记。
4.  点击**合并编辑器解析**，确认合并冲突

### 第 5 步：标记冲突已解决并完成合并

在你手动编辑文件，解决了所有冲突标记后：

- **点击同步更改**

> [!note] 如何尽量避免冲突？
>
> - **频繁地 `pull`**：在开始工作前和提交后，都习惯性地 `git pull` 一下，保持本地代码最新。
> - **沟通**：和队友沟通好谁在修改哪些文件。
> - **小步提交**：频繁地提交小更改，而不是一次性提交大量修改。

## 忽略文件

在主目录（包含 `.git` 文件夹的文件夹）中创建文本文件 `.gitignore`

文件中写入**所有不需要同步**的文件和文件夹，比如

```
.gitignore
.vscode/
.obsidian/
```
